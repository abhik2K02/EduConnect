# Lecture Assistant - Backend Architecture Guide

## Overview
The backend of the Lecture Assistant application is a **Node.js** server built with the **Express.js** framework. It acts as the central brain of the application, orchestrating audio processing, database operations, and vector embeddings for semantic search. A core pillar of this architecture is **Natural Language Processing (NLP)**, which powers the intelligent transcription, summarization, and query capabilities.

## 1. Core Technologies & NLP Integration
- **Express.js** & **Multer**: Web framework and middleware for handling audio file uploads.
- **Firebase Admin SDK**: Secure server-side interactions with Firestore and Firebase Storage.
- **Deepgram API (Speech-to-Text NLP)**: Converts spoken audio into text with high accuracy and low latency.
- **Google Gemini API (Generative NLP / LLMs)**: A Large Language Model used for summarization, intelligent filtering of transcripts, and generating contextual answers (RAG).
- **Pinecone (Semantic NLP)**: A Vector database used to store and query semantic embeddings of lecture transcripts, enabling students to search by meaning rather than just exact keywords.

## 2. The NLP Pipeline (`POST /api/upload-audio`)
This is the heaviest and most critical endpoint. When a teacher finishes recording a lecture, the backend executes a sophisticated **NLP pipeline**:
1.  **Audio Ingestion**: Accepts the audio blob via Multer into memory (`file.buffer`).
2.  **Transcription (STT)**: Sends the buffer to **Deepgram** (`services/deepgramService.js`) to get a raw, highly accurate transcript.
3.  **Intelligent Filtering**: Sends the raw transcript to **Gemini** (`services/geminiService.js`) to filter out filler words, stutters, and off-topic conversations natively using LLM comprehension.
4.  **Generative Summarization**: Analyzes the filtered transcript to generate a concise summary of the lecture's core topics.
5.  **Storage & Database**: Uploads the raw `.webm` audio to Firebase Storage and saves the lecture metadata (URLs, summary, full transcript) to Firestore.
6.  **Semantic Vectorization**: Chunks the filtered transcript into smaller pieces (approx. 1000 characters). Each chunk is mapped to a high-dimensional vector space using Gemini's embedding model (`text-embedding-004`) and upserted into **Pinecone** for future semantic search.

## 3. NLP-Powered Chat Endpoints
### `POST /api/chat` (Retrieval-Augmented Generation)
This endpoint powers the "Chat with Lecture" feature for students using a classic RAG architecture:
1.  **Query Embedding**: Converts the student's natural language question into a semantic vector.
2.  **Semantic Search**: Queries Pinecone with the question vector to retrieve the top 5 most semantically relevant transcript chunks.
3.  **Grounded Generation**: Sends the user's question along with the retrieved transcript chunks (Context) to Gemini to synthesize an accurate, grounded answer based *only* on the lecture material.

### `POST /api/teacher/chat` & `POST /api/teacher/edu-assistant`
These endpoints handle NLP interactions specifically for the teacher:
-   `/api/teacher/chat`: A standalone conversational agent for general teaching advice or lesson planning.
-   `/api/teacher/edu-assistant`: Powers the "Hey Edu" live voice assistant. It takes a voice command (e.g., "Show me a video about photosynthesis") alongside the live transcript context to generate helpful notes and extract search intent on the fly.

## 4. Services Structure (`/services`)
To keep `index.js` clean, the NLP logic is abstracted:
-   `deepgramService.js`: Contains `transcribeAudio()` which handles the STT API call to Deepgram (`nova-2` model).
-   `geminiService.js`: Heavily utilized file housing prompts and API calls to Google's Generative AI for summarization, embedding generation, chat responses, and transcript filtering.
-   `pineconeService.js`: Handles vector upserts and semantic similarity queries.
